# Frontend Generation Prompt — Rental Management System

## Context

You are building the **frontend** for a Rental Management System. A backend already exists, but it was built as a **single-tenant** system (one shared catalog, one Admin tier). This project needs a **multi-tenant marketplace**: any number of Admins, each running their own independent rental store, with Customers browsing a combined marketplace across all of them. That requires backend changes beyond what's currently documented — they're listed below so the team can add them alongside this frontend work. Build the frontend against the model described in this prompt; treat the "Required backend additions" section as the contract to implement/confirm against, not the old single-org docs.

Use the mockups for layout and visual intent only. Never invent a field or endpoint outside what's listed here without flagging it.

---

## Required backend additions (do this alongside/before the frontend work)

The existing API is single-org: every `ADMIN` shares one catalog, one settings resource, and one order queue. To support multiple independent Admins, add an **owning-Admin scope** to every Admin-managed resource:

- **Signup:** allow `role: ADMIN` at self-signup (not just `CUSTOMER`). New Admin accounts are active immediately — no approval step.
- **Product / Variant:** add an `adminId` (owner) set from the authenticated Admin on create. `GET /products` (storefront, public) returns products from **all** Admins with the owning Admin's public store info attached (id, display name); `GET /products?mine=true` (or an equivalent Admin-scoped list) returns only the requesting Admin's own products. Write endpoints (`POST/PUT/DELETE /products/{id}`, variants, images) only succeed for the owning Admin.
- **Attributes, Pricelists, Rental Period templates:** each becomes owned per-Admin instead of shared org-wide — an Admin only sees and edits their own attribute types, pricelists, and period templates. A product can only reference its own owning Admin's attributes/pricelists.
- **Orders / Quotations:** an order's owning Admin is derived from the product(s) in it (assume single-Admin carts/orders for v1 — block checkout if a cart mixes products from two different Admins, or split into per-Admin orders if the team prefers; call this out as a decision needed). `GET /orders` for an Admin returns only orders containing their own products. Confirm/reject/pickup/return/settle/damage-report actions are only permitted for the order's owning Admin.
- **Pickups/Returns schedule, Dashboard, Rental Settings:** all scoped to the requesting Admin's own orders/products — `GET /admin/rental-settings` becomes per-Admin (each Admin sets their own grace period, late-fee %, cap), not a single org-wide config.
- **Public store info:** a lightweight public endpoint (or field on product responses) exposing an Admin's store display name so Customers can see "Sold by [Store Name]" and filter/browse by store.

Everything below assumes these additions exist.

---

## Roles

Exactly two roles — call them **Customer** and **Admin** in the UI and code; do not rename Admin to "Vendor," "Seller," or anything else.

1. **Customer** — browses the marketplace across all Admins, manages their own cart/orders/addresses/invoices.
2. **Admin** — runs their own independent rental store: their own product catalog, attributes, pricelists, rental period templates, quotations, orders, pickups/returns, dashboard, and settings. An Admin never sees or acts on another Admin's products or orders. Any number of Admins can exist; each is a self-contained store owner, not a platform-wide operator.

Signup includes a role choice — Customer or Admin — both self-serve and active immediately on signup. An Admin signup should also capture a store display name (used everywhere a product needs to show who it's sold by).

## Design System (non-negotiable)

- **Layout:** Left-hand vertical sidebar navigation (not a top navbar) for all authenticated views — sidebar holds primary nav (for Admin: Dashboard, Orders, Quotations, Schedule, Products, Pricelists, Attributes, Reports, Settings; for Customer: Home, Products, Orders, Account). Sidebar bottom has a profile icon (→ profile page) directly above a logout button. Duplicate logout in the top-right of every page.
- **Palette:** Restrained, professional balance of light and grey tones. No default Tailwind "indigo/purple SaaS demo" palette. One muted primary color used sparingly for primary actions/status, plus greys for structure.
- **Shape language:** No rounded corners anywhere — buttons, cards, inputs, modals all sharp/square. No emojis in UI copy or icons.
- **Icons:** Font Awesome via CDN.
- **Styling:** Tailwind CSS, used deliberately — no oversized hero sections, no filler illustrations, no placeholder marketing copy. Prioritize real controls, real tables, tight spacing, purposeful whitespace.
- **Structured data:** Render orders, products, pricelist rules, schedules, and deposit ledgers as real sortable/filterable tables — not card grids pretending to be lists.
- **No clutter:** Every screen shows only what that screen's role/state needs.
- **Links:** Every button/link routes to a real destination — no dead links or `#` placeholders.

## Client / Portal (storefront)

- **Home page (signed-in):** Top search bar, cart icon, user avatar → dropdown (Profile, My Orders, Settings, Logout). Left filter rail: Brand/Color/Size etc. from attributes across all Admins, price range, and a **Store/Admin filter** so a Customer can browse one Admin's catalog specifically or the full marketplace. Product grid: image, name, resolved price, stock badge, and a small "Sold by [Store Name]" line on every card.
- **Product detail page:** Image, name, resolved price, "Sold by [Store Name]" (linking to that Admin's storefront view — a filtered product grid for just that store), quantity selector, rental period selector (free-form dates or a template dropdown from that product's owning Admin), "Add to Cart." Variant products open a **Configure modal** with attribute-value pickers (one consistent picker style) before adding. Check availability for the selected variant + dates before allowing add-to-cart.
- **Cart:** Rental period per line, quantity adjust, remove line, "Continue Shopping," proceed to checkout. If the cart contains products from more than one Admin, either block checkout with a clear message ("checkout one store at a time") or visually group lines by store and checkout each group separately — pick one approach and apply it consistently; this depends on the single-Admin-order decision flagged above.
- **Checkout:** Fulfillment method — Standard Delivery (choose/add address) vs Pickup from Store — then review and submit. Submitting sends the order to the owning Admin for approval; no payment step yet. Confirmation screen: order number, status, note that they can pay once approved.
- **My Orders / Order detail:** Orders list shows which store each order belongs to. Status timeline (submitted → approved → paid → scheduled for pickup → picked up → returned → settled → closed, or cancelled/rejected). Actions gated by status:
  - Awaiting approval: Cancel Request.
  - Approved: Pay Now → Razorpay checkout (rental + deposit together). Refetch real order status after checkout closes rather than trusting the client-side success callback.
  - After payment: read-only status, deposit status (held/deducted/refunded) with penalty breakdown if late, invoices with download links.
- **Account area:** Profile with photo, saved addresses (add/edit/remove/set default), password change, order history across all stores.

## Auth

- **Login:** email + password, "Forgot Password" link, inline errors on invalid credentials.
- **Sign up:** first name, last name, email, password, confirm password, terms checkbox, and a **role toggle: Customer / Admin**. If Admin is selected, also collect a store display name. Client-side checks (email format, min length, password match) before submit; final validation trusts the backend.
- **Email verification:** verification-code entry + resend, if shown in the mockups.
- **Forgot / Reset password:** request by email, reset form with new password + confirm, success state.

## Admin Backend

Everything in this section is scoped to the logged-in Admin's own store — never show or allow action on another Admin's data.

- **Global shell:** left sidebar (Dashboard, Orders, Quotations, Schedule, Products, Pricelists, Attributes, Reports, Settings), search bar on list pages, profile + logout, and the store display name visible somewhere in the shell (e.g. sidebar header) so it's always clear which store the Admin is operating.
- **Dashboard:** Active Rentals, Due Today, Upcoming Pickups, Upcoming Returns, Overdue Rentals, Revenue, Deposits Held, Late Fee Collection — for this Admin's own products/orders only, as a clean metrics row deep-linking to a pre-filtered Orders list.
- **Orders — List and Kanban (toggle).** Only this Admin's orders. Status badges across the full lifecycle: awaiting approval, confirmed, paid, scheduled for pickup, active, returned, settled, closed, cancelled, with an overdue indicator layered on active orders past due.
- **Order detail page:** header actions strictly by status, and only for orders this Admin owns:
  - Awaiting approval → Confirm / Reject (with reason).
  - Confirmed → read-only, waiting on customer payment; show invoice.
  - Paid → Schedule Pickup.
  - Scheduled → Confirm Pickup (checklist).
  - Active → Confirm Return (condition notes, damage flag, missing-accessories flag) and Add Damage Report.
  - Returned → Settle (late-fee calc, deposit deduction/refund, stock update, invoice generation together).
  - Settled/Closed → read-only, deposit ledger and invoice history.
  Order lines table: product, variant, quantity, unit, unit price, amount, running subtotal/total footer.
- **Invoices:** listed per order with download links, generated automatically at confirmation and again at settlement if a late fee applies — no manual create/send/print flow.
- **Quotations (in-store):** this Admin's own quotations only — list by status, create for a customer from a template, edit lines while editable, Confirm (converts to a confirmed order) or Reject.
- **Quotation templates:** this Admin's own — name, header, footer, default terms.
- **Product creation flow:** name, description, category, price, fixed security deposit amount, images, Attributes & Variants tab (this Admin's own attribute types/values; variants = attribute-value combos each with their own stock quantity — skip this tab for products with no variant axes), active/inactive toggle (soft-delete only).
- **Rental Periods:** this Admin's own named templates (e.g. "Weekend," "1 Week," "1 Month") — simple list/create/edit/delete.
- **Schedule page:** this Admin's own day's pickups and returns, with quick links into each order.
- **Pricelist page:** this Admin's own pricelists — validity window (blank = default/unbounded), per-rule pricing (product/variant + duration + flat unit price). Default pricelist can't be deleted; when more than one could apply, oldest wins.
- **Attributes page:** this Admin's own attribute types and values.
- **Reports:** revenue and rental-activity charts for this Admin's own store.
- **Settings:** this Admin's own rental settings — grace period, daily late-fee percentage, maximum late-fee multiplier, default pickup/return windows. Per-Admin, not shared.

## Business logic to reflect in the UI

- **Security deposit:** fixed amount per product, collected with the rental payment at checkout-after-approval. Held until return; on-time return refunds it in full; late return deducts a calculated penalty up to the full deposit, with any excess shown as an outstanding late-fee invoice. Show deposit status and penalty breakdown from the ledger — never recalculated on the frontend.
- **Late fees:** grace period, then a daily compounding percentage of the product's price, capped at a multiple of that price — configured per-Admin in Settings, shown as a calculated result on orders/invoices, never recomputed client-side.
- **Pickup & Return:** explicit Confirm Pickup / Confirm Return actions with checklist/condition-notes/damage fields on the order detail page.
- **Marketplace scoping:** every Admin-side screen and action is implicitly filtered to "my store" — there is no cross-Admin visibility anywhere in the Admin backend. Customers, by contrast, always see the combined marketplace unless they've applied a store filter.

## Technical requirements

- Use the existing frontend stub's routing, JWT interceptor, and installed dependencies — extend, don't rebuild auth handling. The JWT should carry both role and (for Admins) their own id for scoping — confirm this is in the token claims once the backend additions above are in place.
- Add the Razorpay Checkout.js script for the payment step; otherwise add only libraries genuinely needed.
- Tailwind + Font Awesome CDN as specified above.
- Fetch against real API endpoints/response shapes — no hardcoded sample data left in the final code except as an explicit loading placeholder.
- Handle loading, empty, and error states on every data-driven screen.
- Always display server-computed totals, prices, and fees as-is — no float arithmetic on money client-side beyond display formatting.
